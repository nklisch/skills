# Cross-vendor `.agents/rules/` auto-load mechanism

## Brief

Today the agile-workflow plugin's dense agent rules live inside the managed
`<!-- agile-workflow:start -->` section of `AGENTS.md`, and reusable patterns
produced by `gate-patterns` live in `.agents/skills/patterns/` — meant to
"auto-load when implementing, designing, verifying, or reviewing." In practice
the patterns skill rarely loads, because skill auto-load is keyword/heuristic
and unreliable. The legacy Claude-only `.claude/rules/*.md` mechanism, by
contrast, reliably force-fed rules into every session — but it is Claude-only
and `convert` deprecates it.

This epic introduces a **generic `.agents/rules/` directory** that a plugin hook
force-loads into context on coding work, reliably, in **both Claude Code and
Codex**. It is the portable replacement for `.claude/rules/`. Producers
(`convert`, `gate-patterns`, or the user) drop `*.md` files into `.agents/rules/`;
the hook is content-agnostic and injects them. The dense behavioral rules move
**out of** the always-loaded `AGENTS.md` section (de-duplication) into
`.agents/rules/agile-workflow.md`, and `gate-patterns` writes a concise patterns
digest into `.agents/rules/patterns.md` while the detailed pattern bodies stay in
the `.agents/skills/patterns/` skill (the digest points there for detail).

The same effort hardens `convert`'s legacy cleanup so it can **never lose data**:
every legacy artifact must have a verified replacement at its canonical
destination before any destructive step.

## Strategic decisions

- **AGENTS.md keeps dense pointers, not bare orientation** (user decision): the
  managed AGENTS.md section stays in its dense-pointer style — substrate
  orientation + `work-view` query patterns + grep-able pointers to the canonical
  rules file `.agents/rules/agile-workflow.md`, the patterns skill, and a MANDATORY
  "read `.agents/rules/*.md` before designing/implementing/reviewing" directive. The
  *dense behavioral rule prose* (tag semantics, test integrity, advisory-review,
  refactor-conventions) moves into `.agents/rules/agile-workflow.md`. — Rationale:
  AGENTS.md always loads and points reliably at the rules; the hook force-loads the
  rules themselves at session start.
- **Patterns ↔ rules relationship**: `gate-patterns` writes a concise
  `.agents/rules/patterns.md` digest (slug + one-liner + "load the `patterns`
  skill / read `.agents/skills/patterns/<slug>.md` for full detail"); the
  detailed pattern bodies stay in `.agents/skills/patterns/`. The hook is generic
  — it injects whatever `*.md` files exist in `.agents/rules/`. — Rationale:
  keeps the patterns skill so existing consumers (`gate-cruft`, `gate-docs`,
  `refactor-design`, `implement-orchestrator`) are unchanged; satisfies both
  "generic `.agents/rules` the hook leverages" and "load patterns skill for
  detail"; digest + detail are generated in one `gate-patterns` pass so they
  cannot drift.
- **Graceful degradation**: AGENTS.md retains the orientation + pointer so that
  even when the hook does not fire (no substrate, untrusted hook in Codex,
  non-coding session) agents still know where rules live.
- **Foundation-doc roll-forward is deferred** to the docs child feature and lands
  *with* the implementation, not at scope time — the mechanism does not exist
  yet, and rolling-foundation requires docs to describe the present.
- **Trigger: primary firing on `SessionStart` + `PostCompact`, with a coding-prompt
  fallback** (user decision, revising the earlier first-coding-prompt-only choice):
  emit `.agents/rules/` at session start and after compaction (mirrors legacy
  `.claude/rules/`); a broad coding-prompt `UserPromptSubmit` detector — separate
  from the workflow-snapshot gate (`cheap_action_candidate`/`is_actionable`, which
  would miss "fix failing tests", "continue", "debug this build error", file edits)
  — is the once-per-session fallback. All paths share per-epoch + content-hash dedup.
- **Post-compaction re-injection is guaranteed** (resolves Codex finding 4):
  `SessionStart`/`PostCompact` now emit `.agents/rules/` directly (not just bump the
  epoch), so rules reload after compaction even during auto-continuation with no
  user prompt.
- **`.agents/rules/patterns.md` is a generated digest with a banner + source hash**
  (Codex review): `.agents/skills/patterns/` stays the single source; the digest
  carries a "generated by gate-patterns — do not hand-edit" banner and a hash of the
  source so drift is detectable. Codex's single-source alternative (hook reads the
  patterns skill index directly) was set aside to honor the "generic `.agents/rules/`
  the hook leverages" direction; banner+hash mitigates duplication.
- **Graceful degradation is enforced, not assumed** (Codex review): the slim
  AGENTS.md block carries a MANDATORY "read `.agents/rules/*.md` before
  designing/implementing/reviewing" directive (AGENTS.md always loads), AND the
  agile-workflow design/implement/review skills' grounding phases are updated to read
  `.agents/rules/*.md`. This is its own child feature so load-bearing rules (test
  integrity, tag semantics) never vanish when the hook does not fire.
- **Slimming the AGENTS.md section is itself a data-loss vector** (Codex review):
  `convert --update` overwrites the managed section, so the extraction feature must
  write AND verify `.agents/rules/agile-workflow.md` BEFORE removing dense content
  from AGENTS.md — the content-integrity gate applies to the slim, not just to legacy
  cleanup.

## Other agent review (Codex via peeragent, 2026-05-31)

One focused cross-model advisory pass (Codex, effort high, read-only). Accepted
findings, folded into the decisions above and the decomposition below:
- Data-safety must also cover the AGENTS.md *slim/overwrite*, not just legacy
  `.claude/rules/*` cleanup — `convert --update` overwrites the managed section.
  → write+verify `.agents/rules/agile-workflow.md` before slimming (convert-extract);
  generalized content-integrity gate (convert-safety).
- Graceful degradation is weaker than assumed — many skills read AGENTS.md, not
  `.agents/rules/*`. → mandatory read-directive in the slim block + skill grounding
  updates (new skill-grounding feature).
- The rules trigger must not reuse the workflow-snapshot gate; use a broader coding
  detector (hook feature).
- Post-compaction re-injection only happens on the next UserPromptSubmit; document
  the contract (hook feature).
- Keep `.agents/skills/patterns/` single-source; `.agents/rules/patterns.md` is a
  generated digest with banner + hash (patterns-digest feature).
- Convert needs a block-level preservation manifest for split files: classify each
  source block, route+hash-verify or preserve-in-place as ambiguous, before any
  shim/removal (convert-safety feature). Note: legacy non-pattern rule prose
  routes to a user-owned `.agents/rules/<name>.md` (default `project.md`) — NEVER
  `patterns.md` (reserved for the generated digest) or `agile-workflow.md`
  (plugin-managed).

(consumer) lands first; the producers (gate-patterns digest, convert extraction) and
the skill-grounding update parallelize after it; convert's data-safety gate builds on
the extraction (same file, serialized); docs sync lands last to describe the new

### Decomposition risks

- `convert-extract` and `convert-safety` both edit `convert/SKILL.md`; the depends_on
  edge serializes them to avoid contention. Boundary: extract owns the slim +
  write-verify of `agile-workflow.md`; safety owns the generalized gate + manifest for
  legacy artifacts.
- `skill-grounding` touches several skill files with a one-line grounding addition
  each; risk is missing a skill — enumerate the design/implement/review family
  explicitly during its design pass.
- The hook's broad coding detector risks over-firing. Tune against the Codex
  false-negative examples; cap by bytes and dedup per epoch.

## Convert legacy-cleanup data-safety findings (audit, 2026-05-31)

Root cause: `convert` trusts *delegation to owning skills* + *reference-integrity
(pointer rewriting)*, but never **verifies content reached its canonical
replacement** before a destructive step (`git rm` / shim / symlink-replace).
Fix theme for feature 4: add a **content-integrity gate** distinct from
reference-integrity — confirm the replacement exists and holds the content (both
halves for split-destination files) before any destructive action.

1. **No content-migration verification before destroying `patterns.md`**
   (Phase 7 ~473-501; Sync S3 ~834-839). DATA-LOSS.
2. **`patterns.md` is a split-destination file** (structural → `.agents/skills/patterns/`,
   prose → AGENTS) with no defined split algorithm and no guard that BOTH halves
   were preserved. DATA-LOSS.
3. **Reference-integrity rewrites pointers only, not content** (Phase 1.8 ~215-222) —
   a pointer can be repointed while the content is never migrated. PARTIAL-LOSS.
4. **`drift_user`/ambiguous** has no atomic confirm→import→verify→shim sequence;
   can strand or drop content (Phase S1 ~755-763, S3 ~834-844). FRAGILE.
5. **`cleanup_scope: generated-only`** can shim `patterns.md` when it matches an
   old template, with no content check — needs a `patterns.md` carve-out
   (Phase 7 ~486-489). DATA-LOSS.
6. **Bespoke-skill convergence** folds patterns "per gate-patterns Phase 1" with
   no verification the import succeeded before removing the source (Phase 8.6
   ~612-638). DATA-LOSS.

## Verified hook contract (Claude Code + Codex, 2026-05-31)

Both ecosystems share the contract the existing hook already uses:
- Events: `SessionStart`, `UserPromptSubmit`, `PostToolUse`, `PostCompact`, etc.
- `hooks.json`: `{hooks:{<Event>:[{matcher,hooks:[{type:"command",command}]}]}}`.
- Output: JSON `hookSpecificOutput.additionalContext` (string) injected as context;
  plain stdout also accepted.
- Env: Codex exports `PLUGIN_ROOT`/`PLUGIN_DATA` AND `CLAUDE_PLUGIN_ROOT`/`CLAUDE_PLUGIN_DATA`;
  existing `${PLUGIN_ROOT:-${CLAUDE_PLUGIN_ROOT}}` is already portable.

Codex caveats: `additionalContext` renders as a *visible* developer message
(openai/codex#16933); no built-in per-session dedup yet (#21675) → the script
must self-dedup (it already does, via the epoch state file); `PreToolUse` does
NOT support `additionalContext` (#19385); plugin hooks require one-time user
trust on install.

The existing `hooks/scripts/prompt-context.py` already fires on
`UserPromptSubmit` with per-session, per-epoch dedup (`SessionStart` resets,
`PostCompact` bumps the epoch), prompt-gating, and substrate-gating — the rules
loader extends this machinery (the `code_design`/`advisory_review` capsule
triggers are a good "coding prompt" proxy).

## Trigger decision (updated with user, 2026-05-31)

Hybrid firing, mirroring legacy `.claude/rules/`:
- **Primary**: emit `.agents/rules/*.md` at `SessionStart` and `PostCompact`
  (once per session, again after compaction), unconditionally — substrate + flag
  gated, no prompt to gate on.
- **Fallback**: a broad coding-prompt `UserPromptSubmit` detector emits the rules
  once per epoch if the session-start emission did not happen (substrate appeared
  mid-session, host skipped SessionStart, or content changed).
- Shared per-epoch + content-hash dedup → rules load exactly once per session.
- AGENTS.md keeps dense pointers to `.agents/rules/agile-workflow.md` + the
  patterns skill; `.agents/rules/patterns.md` carries the "load the patterns skill
  for detail" pointer.

(Supersedes the earlier first-coding-prompt-only decision.)
