# Suggestion & Design Output Formats

## Suggestion Format (Phase 2: Provoke)

Present each suggestion as:

```markdown
### {Provocative Name}

**Lens:** {Elimination | Unification | Inversion | Algebraic | Declarative | Domain Crystallization}

**Thesis:** {One sentence — what's the insight and why does it matter}

**Affected code:**
- `src/path/to/file.ts` — {what changes}
- `src/path/to/other.ts` — {what changes or disappears}

**What gets simpler:**
- {Concrete benefit 1 — e.g., "400 lines collapse to 60"}
- {Concrete benefit 2 — e.g., "Bug fixes happen once instead of in 5 places"}

**What it costs:**
- {Risk or difficulty 1 — e.g., "Requires migrating all callers of X"}
- {Risk or difficulty 2 — e.g., "Breaks the existing plugin API"}
```

### Good Suggestion Names

Names should be a thesis, not a task:
- "Auth is just middleware composition" (not "Refactor auth module")
- "These five parsers are one parser with five configs" (not "Deduplicate parsers")
- "The state machine that doesn't know it's a state machine" (not "Add state machine")
- "Delete the plugin system" (not "Simplify plugin architecture")

### Suggestion Quality Checklist

Before presenting, verify each suggestion:
- [ ] Names a specific conceptual lens
- [ ] Could NOT be generated by "refactor this code" prompt to a generic LLM
- [ ] Would surprise a senior engineer — then convince them
- [ ] States concrete impact (lines removed, concepts unified, indirection eliminated)
- [ ] Acknowledges real costs, not just benefits

## Design Document Format (Phase 4: Design)

After discussion converges, produce a design document matching the format consumed by
the `implement` and `implement-orchestrator` skills:

```markdown
# Design: {Refactoring Name}

## Overview
{What this refactoring achieves. Reference the conceptual lens(es) that drove it.
State the core insight in one sentence.}

## Implementation Units

### Unit 1: {Name}

**File**: `src/path/to/file.ts`

\`\`\`typescript
// Exact interfaces, types, and function signatures
// that will be created or modified
\`\`\`

**Implementation Notes**:
- {Why this approach, not just what}
- {Non-obvious decisions or tricky parts}
- {What existing code this replaces and how callers migrate}

**Acceptance Criteria**:
- [ ] {Testable assertion}
- [ ] {Testable assertion}

---

### Unit 2: {Name}

**File**: `src/path/to/other.ts`

\`\`\`typescript
// Exact interfaces, types, and function signatures
\`\`\`

**Implementation Notes**:
- {Notes}

**Acceptance Criteria**:
- [ ] {Testable assertion}

---

### Unit N: Migration & Cleanup

**Files**: `src/path/to/caller1.ts`, `src/path/to/caller2.ts`, ...

**Implementation Notes**:
- {How to migrate callers from old API to new}
- {What files to delete after migration}
- {Verification that nothing references the old code}

**Acceptance Criteria**:
- [ ] All callers migrated to new API
- [ ] Old files/exports deleted
- [ ] No dead code remains

---

## Implementation Order
1. {Unit to implement first — dependencies resolved}
2. {Next unit — depends on unit 1}
3. {Migration unit — always last}

## Testing
### Unit Tests: `tests/path/file.test.ts`
{Test structure, key test cases, edge cases to cover}

## Verification Checklist
{Commands to run: build, test, lint, typecheck}

## Rollback Plan
{How to revert if the refactoring causes problems.
What intermediate states are safe to stop at.}
```

### Design Quality Checklist

- [ ] Every implementation unit has exact file paths and code signatures
- [ ] Types/interfaces are in the project's language, not prose descriptions
- [ ] Implementation order resolves dependencies
- [ ] Migration unit exists for any API changes
- [ ] Acceptance criteria are testable assertions, not subjective judgments
- [ ] An implementer agent can write code from this without asking questions
- [ ] Rollback plan identifies safe stopping points
